协调器模式

发表于 2019-12-10

协调器模式（Coordinator Pattern）

协调器模式是一种结构型设计模式，用来组织viewcontroller之间的跳转逻辑。它包含以下部分

  1. coordinator：定义了所有concrete coordinators必须实现的方法和属性。具体来说，它定义关系属性，子级和router。它还定义了呈现方法，present和dismiss。通过持有coordinator protocol而不是直接持有concrete coordinators，你可以将父协调器与其子协调器解耦。这也使父协调员可以在单个属性中保留各种具体的子协调员，即children
  2. concrete coordinator：实现了coordinator protocol。它知道如何创建具体的view controller以及view controller的显示顺序。
  3. router：一种协议，定义了所有 concrete router必须实现的方法。具体来说，它定义了用于显示和关闭视图控制器的present和dismiss方法
  4. concrete router：知道如何显示view controller，但不知道在显示什么或接下来将显示哪个view controller。相反，coordinator告诉router要显示哪个view controller
  5. concrete view controllers：mvc中的view controller，他们不知道其他的view controller。他们是coordinator的代理。

责任链模式

发表于 2019-12-09

责任链模式（Chain-of-Responsibility Pattern）

责任链模式是一种行为模式，它让事件被多个处理器中其中一个处理。包含以下3部分：

  1. client：接受事件并将事件传递给处理 handler protocol的实例。事件可以是简单的，仅具有属性的结构，也可以是复杂的对象，例如复杂的用户操作。
  2. handler protocol：定义了handler必须实现的属性和方法。也可以使用基类
  3. concrete handler：实现handler protocol中的方法，被client直接持有。收到事件后，它首先尝试处理它。如果无法执行此操作，则将事件传递给下一个handler

命令模式（Command Pattern）

命令模式是一种行为模式，将执行操作的信息封装到命令对象中。它包含以下3个类型：

  1. invoker：储存和执行命令
  2. command：将行为封装成对象
  3. receiver：command所作用的对象

组合模式（Composite Pattern）

组合模式是一种结构型设计模式，将一组对象组合成树状结构，以便可以像对待一个对象一样来对其进行操作，它包含以下三个类型：

  1. component protocol：确保树中的所有构造都可以以相同的方式处理
  2. leaf：树的组成部分，没有子元素
  3. composite：持有leaf对象和composites的容器

中介者模式

发表于 2019-11-29

中介者模式（Mediator Pattern）

中介者模式是一种行为模式，封装了对象间的通信方式，它包含了四个类型：

  1. colleagues：需要互相交流的对象。他们实现了colleague protocol。
  2. colleague protocol：定义了每个colleague必须实现的方法和属性。
  3. mediator：控制colleague之间交流的对象。它实现了mediator protocol。
  4. mediator protocol：定义了mediator必须实现的方法和属性。

每个colleague都通过mediator protocol持有mediator的引用。每个colleague都通过mediator来相互交流，而不是直接交流。

每个colleague都可能通过mediator发送或接收消息。

跟多播委托模式相似，但是有不同之处

享元模式

发表于 2019-11-29

享元模式（Flyweight Pattern）

享元模式是一种结构性设计模式，可以最大程度地减少内存使用和处理。

此模式提供的对象均共享相同的基础数据，从而节省了内存。它们通常是不变的。

享元模式具有称为flyweights的对象，以及返回它们的静态方法

享元模式是单例模式的变体，在享元模式中，通常有多个相同类的不同对象。

外观模式（Facade Pattern）

外观模式是一种结构模式，给一个复杂的系统提供一键简单的接口，它包含两种类型：

  1. facade：提供了简单的接口来跟系统交互，这允许开发者使用外观，而不是了解系统中的多个类并与之交互
  2. dependencies：被facade持有，每个dependency负责一个巨大任务中的一部分

发表于 2019-11-28

多播委托模式（Multicast Delegate Pattern）

多播委托模式一种行为模式，是代理模式的变体。它能让你创建一对多的代理关系，而普通的代理模式只有一对一的关系。它包含以下四个类型：

  1. object needing a delegate(delegating object)：持有一个或多个代理的对象
  2. delegate protocol：定义了一个代理需要实现的方法
  3. delegate(s)：实现了delegate protocol的多个对象
  4. multicast delegate：持有多个代理的辅助类，能在有事件发生时通知每个代理

代理模式和多播委托模式做大的区别就在multicast delegate的存在。

状态模式

发表于 2019-11-27

状态模式（State Pattern）

状态模式是一种行为模式，能让对象通过修改他们的“状态”，在运行时改变自己的行为。“状态”是指描述给定对象在给定时间如何行为的数据集。

状态模式包含以下三种类型：

  1. context：具有当前状态且其行为会发生变化的对象。

  2. state protocol：定义了必需的方法和属性。开发人员通常用基类代替协议。这样，他们就可以在基类中定义属性。

    即使使用了基类，也不能直接实例化它。相反，它的定义仅是为了能够被子类化。

  3. concrete state：遵循state protocol或继承自基类。context会持有它的当前状态，但不知道具体的状态类型。context会使用多态来改变行为：concrete state定义了context的行为。如果你想要一个新的行为，定义一个新的concrete state。

应该在哪里修改当前状态呢？是在context里修改？还是在concrete state修改？还是在其他什么地方修改呢？状态模式并没有告诉你应该在哪里写状态修改逻辑，这视实际情况决定。

什么时候应该使用状态模式

使用状态模式来创建一个系统，该系统在其生命周期内会在两个或多个状态之间进行更改。状态可以限制数量（“封闭”集），也可以不受限制（“开放”集）。例如，可以使用一组封闭的“交通信号灯状态”来定义交通信号灯。在最简单的情况下，它会从绿色变为黄色，再变成红色，最后变回绿色。

一个动画引擎可以定义为一组开放的“动画状态”。它具有无限的旋转，平移和其他动画。

开放式和封闭式的状态模式的都使用多态来更改行为。结果，您通常可以使用此模式来消除switch和if-else语句。

无需在context中管理复杂的状态，而是将调用传递到当前状态中。如果你有一个包含多个switch或if-else语句的类，请尝试使用状态模式来定义它。这样你就可能会创建一个更灵活，更容易维护的系统。

状态模式和策略模式的区别

策略模式更多的是解决不同状态下，单行为的变化；而状态模式则是解决不同状态下多行为的变化。

适配者模式

发表于 2019-11-26

适配者模式（Adapter Pattern）

适配者模式是一个行为模式，能够让不兼容的类型一起工作，它包含以下4个部分：

  1. object using an adapter：依赖new protocol的对象
  2. new protocol：使用所需的协议
  3. legacy object：在制定协议之前就已经存在，并且不能直接修改来遵循协议
  4. adapter：遵循协议，并将调用传递到legacy object上

新iPhone上没有耳机插口，如果你想把你的耳机插进新iPhone里，就需要一个适配者来连接彼此

什么时候应该使用适配者

类，模块和方法并不可能总是能够修改的，尤其是当他们来自第三方库时，所以有时你不得不适配它们。

你可以通过扩展一个存在的类或者新建一个类来创建一个适配者。
